# 活動報名系統 PRD (Product Requirements Document)

## 1. 專案概述
### 背景與動機
傳統活動報名往往依賴紙本、零散的線上表單或人工確認，容易導致資料遺失、對帳困難、現場報到混亂等問題。本專案旨在開發一個整合化的雲端活動管理平台，簡化從活動發布、名額控管到現場簽到的完整流程，提升主辦方的行政效率並優化參加者的報名體驗。

### 目標用戶
- **活動主辦者**：需要發布活動、管理報名表單、審核參加者與檢視報名數據的人員或組織。
- **一般參加者**：對活動有興趣，需要進行報名、繳費、取得票券並於現場簽到的使用者。

### 核心價值主張
- 簡化活動籌備與辦理流程，降低人工處理成本。
- 提供參加者直覺、無縫的報名與報到體驗。
- 透過數據儀表板，協助主辦方掌握活動成效與參加者輪廓。

## 2. 功能需求
### 主要功能與使用者故事

#### 1. 動態報名表單編輯器 (Dynamic Form Builder)
- **描述**：主辦者可根據活動需求，自定義報名欄位。支援文字輸入、單選/多選、下拉選單、檔案上傳（如學生證影本）。提供預覽功能，確保表單邏輯正確。
- **使用者故事**：
  - 作為 **活動主辦者**，我希望 **能自由新增與編輯表單欄位（如文字、選擇題、檔案上傳）**，以便 **精準收集參加團隊或活動所需的不同資訊**。
  - 作為 **活動主辦者**，我希望 **能預覽報名表單**，以便 **確保報名頁面的顯示效果與邏輯正確**。

#### 2. 名額管理與自動候補系統 (Capacity & Waitlist Management)
- **描述**：即時監控剩餘名額，並在滿額時啟動候補機制。當名額全滿時，後續報名者將自動進入「候補名單」；若正取者取消，系統將依序自動發送遞補通知，並限時完成確認。
- **使用者故事**：
  - 作為 **活動主辦者**，我希望 **系統能自動控管名額上限與候補名單**，以便 **節省人工確認名額與聯絡候補者的時間**。
  - 作為 **一般參加者**，我希望 **在額滿時能自動登記候補，並在正取取消時收到遞補通知**，以便 **把握參加活動的機會**。

#### 3. 電子票券與 QR Code 簽到 (Digital Ticketing & QR Check-in)
- **描述**：報名成功後，系統自動產生唯一的 QR Code 電子票券並寄送至用戶 Email。活動現場工作人員可透過行動裝置掃描票券，即時完成簽到。
- **使用者故事**：
  - 作為 **一般參加者**，我希望 **報名成功後能收到專屬的 QR Code 票券**，以便 **在活動當天快速報到入場**。
  - 作為 **活動主辦者**，我希望 **能用手機掃描參加者的 QR Code**，以便 **即時完成簽到並自動同步報到數據**。

#### 4. 多管道自動化通知 (Automated Notification System)
- **描述**：系統在關鍵節點自動發送訊息，包含「報名成功確認」、「繳費提醒」、「活動前 24 小時通知」以及「活動變更/取消公告」。支援 Email、簡訊或 App 推播。
- **使用者故事**：
  - 作為 **一般參加者**，我希望 **在報名、繳費及活動前能收到自動通知**，以便 **隨時掌握自身報名狀態及活動資訊，避免錯過重要進度**。
  - 作為 **活動主辦者**，我希望 **系統能自動發送重要通知給所有參加者**，以便 **減少人工聯絡的龐大負擔與遺漏風險**。

#### 5. 視覺化數據分析 Dashboard (Data Analytics Dashboard)
- **描述**：提供主辦者直觀的決策依據。以圖表形式顯示報名人數成長趨勢、參加者背景分佈（如科系、年齡）、以及報到率分析。支援一鍵匯出完整的 Excel 名單。
- **使用者故事**：
  - 作為 **活動主辦者**，我希望 **能透過視覺化圖表查看報名趨勢與參加者背景**，以便 **快速評估行銷成效並調整策略**。
  - 作為 **活動主辦者**，我希望 **能一鍵匯出報名名單為 Excel 格式**，以便 **進行後續的離線處理或核對工作**。

## 3. 非功能需求
### 技術限制
- **開發框架**：選擇 Flask (後端) + Jinja2 + 前端技術 (HTML, CSS, JavaScript)
- **資料庫**：SQLite (適合初期 MVP 輕量化部署，後續可視流量升級為 PostgreSQL 或 MySQL)

### 效能與安全考量
- **防呆與驗證**：前端與後端皆需實作表單驗證，確保報名資料格式正確，並防範 XSS 或 SQL Injection 攻擊。
- **高併發防護**：考量搶票可能遇到的並行請求，需實作資料庫的 Transaction 與 Row Level Lock 避免超賣名額 (Overbooking) 問題。
- **敏感資料保護**：參加者個資與上傳之檔案應有適當的權限管控與加密保護；密碼需經安全格式 (如 bcrypt) 加密。

## 4. MVP 範圍
### Must Have (必須具備)
- 會員註冊與登入功能 (主辦者與參加者)。
- 建立活動與基本設定 (時間、地點、名額上限、圖文說明)。
- 基本報名表單填寫與報名者名單後台管理。
- 電子票券產生與 QR Code 現場掃描簽到 (主辦方可透過手機操作)。
- 基礎 Email 自動通知（報名成功通知）。

### Should Have (應該具備)
- 動態報名表單編輯器 (自定義報名收集欄位)。
- 自動候補名單與缺額遞補機制。
- 視覺化報名數據分析 Dashboard 與 Excel 名單匯出。

### Nice to Have (有了更好)
- 檔案附件上傳功能 (如學生證等附件)。
- 多管道通知支援 (如簡訊、App 推播)。
- 第三方金流 API 串接 (購票/繳費功能)。
- 針對一般參加者的歷史報名歷程與電子票夾介面。
